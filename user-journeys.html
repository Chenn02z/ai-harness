User Journey Map

AI Harness now starts with product interrogation, not wishful setup.

This page is the maintained visual map of what an indie developer can do with the harness and which path should happen first. The intended first move after a fork is $bootstrap: grill the founder, replace the template product backbone, and leave the repo in a state where normal milestones and specs are worth writing.

Primary Journeys

1. Bootstrap a fresh fork into a real product workspace

The developer has copied the harness but has not yet clarified what they are actually building.

Ready
  • Start with $bootstrap.
  • Keep asking until target user, pain, outcome, first surface, launch slice, non-goals, and constraints are concrete enough to write down.
  • Populate docs/PRODUCT.md, docs/CONTEXT.md, project-local AGENTS.md details, and an initial milestone when appropriate.

2. Turn vague work into an Accepted milestone

The developer has product direction, but a feature or initiative still needs shape and scope discipline.

Ready
  • Use $requirements for milestone framing and blocker resolution.
  • Use $grill-with-docs when terminology or scope starts drifting.
  • Do not mark the milestone Accepted while blocking questions remain.

3. Convert an Accepted milestone into an executable spec

The developer wants a contract precise enough that implementation can be judged, not merely narrated.

Ready
  • Use $spec only from an Accepted milestone.
  • Keep failure modes, acceptance criteria, and verification explicit.
  • Hand the spec to $dev-loop only after the spec itself is Accepted.

4. Ship through the multi-agent development loop

The developer wants disciplined exploration, editing, testing, and review instead of one broad prompt.

Growing
  • Use $dev-loop for implementation, $test for scoped verification, and reviewer feedback before calling the work done.
  • Keep the main agent as coordinator and reconciler, not the silent do-everything worker.
  • Missing project-specific commands should be documented as maturity gaps, not hidden.

5. Keep docs, skills, and agents aligned as the project evolves

The developer needs durable terminology and workflow updates after real decisions settle.

Ready
  • Use $context when product direction, workflow boundaries, or role definitions change.
  • Prefer surgical doc updates over sprawling rewrites.
  • Keep implementation details out of docs/CONTEXT.md.

6. Audit what happened and show evidence

The developer wants proof of process, not only final prose summaries.

Growing
  • Use $codex-agent-tracer for non-trivial work.
  • Treat .agent-trace/ as local evidence and durable handoffs as the compact summary.
  • Use review artifacts like the product evaluation HTML to improve the next product-facing path.

What Users Can Do

Founding

Interrogate a product idea

Use $bootstrap to force a founder-quality answer set before normal planning begins.

Planning

Create durable milestone contracts

Use $requirements to turn fuzzy goals into scoped milestones with blockers and acceptance criteria candidates.

Specification

Write implementation-grade specs

Use $spec to convert Accepted milestones into executable contracts for development.

Implementation

Run a role-separated dev loop

Use $dev-loop plus the project agents to explore, edit, verify, and review with explicit ownership.

Governance

Pressure-test plans and terminology

Use $grill-with-docs when ambiguity, scope creep, or naming drift threatens trust in the docs.

Alignment

Keep context synchronized

Use $context to keep docs, skills, and agent rules coherent after decisions settle.

The harness is strongest when it is used as a strict workflow spine, not as a replacement for the actual product codebase.

Key Artifacts

Startup contract

AGENTS.md defines non-negotiable operating rules and is where project-specific command gaps should be recorded.

Product backbone

docs/PRODUCT.md and docs/CONTEXT.md become product-specific during bootstrap and stay durable afterward.

Execution evidence

.agent-trace/ holds local traces, while specs, milestones, and reviews preserve the durable summary.
