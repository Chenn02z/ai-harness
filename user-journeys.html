Developer Journeys

1. Start from the template and bootstrap a real product

The developer has just forked and needs to turn generic harness docs into a product backbone.

Ready
  • Run $bootstrap to go through an adversarial founder interview.
  • Answer the grilling until docs/PRODUCT.md, docs/CONTEXT.md, and docs/ARCHITECTURE.md reflect the real product.
  • Determine the MVP boundary — what ships first, what's post-MVP.
  • Derive architecture seams from that boundary, not from upfront speculation.
  • End with an MVP milestone ladder and bare future milestones.

2. Turn a fuzzy request into a durable milestone

The developer has a product direction but the scope isn't decided yet.

Ready
  • Use $requirements to turn vague intent into an Accepted milestone contract.
  • Reference docs/ARCHITECTURE.md to respect seams and deferred work.
  • Every milestone includes an ## MVP Deliverable — what the user sees or does when it ships.

3. Write a spec that fits into the architecture

The developer needs an executable spec that preserves the long-term architecture without overbuilding.

Ready
  • Use $spec to convert an Accepted milestone into implementation contracts.
  • Every spec declares its ## Architecture Reference and ## Architecture Seams.
  • The spec IS the MVP path — no version churn, just what's needed now.

4. Implement with role-separated agents

The workflow must delegate exploration, implementation, verification, and review to distinct agents.

Ready
  • Use $dev-loop to orchestrate explorer → implementer → $test → reviewer.
  • Before implementation, surface architecture seams — what boundaries must be built to extend cleanly later.
  • Keep the main agent as coordinator and reconciler, not the silent do-everything worker.
  • Missing project-specific commands should be documented as maturity gaps, not hidden.

5. Replan after a phase ships

The developer has completed all milestones in a phase and wants to shape the next increment based on what actually shipped.

Growing
  • Manually trigger plan-next when a phase is complete.
  • plan-next reads completed milestones, product direction, architecture seams, and bare future milestones.
  • Proposes the next phase — how many milestones and at what granularity — and produces Draft milestones.
  • Hands Draft milestones to $requirements only; may reshape or discard bare future milestones.

6. Keep docs, skills, and agents aligned as the project evolves

The developer needs durable terminology and workflow updates after real decisions settle.

Ready
  • $context auto-triggers after settled decisions — milestones completed, specs Verified, seams opened or closed.
  • Updates docs/ARCHITECTURE.md when seams open or close.
  • Updates user-journeys.html and README.md when workflow paths or skill names change.
  • Prefer surgical doc updates over sprawling rewrites.
  • Keep implementation details out of docs/CONTEXT.md.

7. Audit what happened and show evidence

The developer wants proof of process, not only final prose summaries.

Growing
  • Use $codex-agent-tracer for non-trivial work.
  • Treat .agent-trace/ as local evidence and durable handoffs as the compact summary.
  • Use review artifacts like the product evaluation HTML to improve the next product-facing path.

What Users Can Do

Founding

Interrogate a product idea

Use $bootstrap to force a founder-quality answer set before normal planning begins. Derives architecture seams from the MVP boundary.

Planning

Create durable milestone contracts

Use $requirements to turn fuzzy goals into scoped milestones with MVP deliverables, blockers, and acceptance criteria.

Architecture

Document current structure and seams

Use docs/ARCHITECTURE.md to track approved seams and deferred architecture — lightweight boundaries, never speculative frameworks.

Specification

Write implementation-grade specs

Use $spec to convert Accepted milestones into executable contracts that reference architecture seams.

Implementation

Run a role-separated dev loop

Use $dev-loop plus the project agents to explore, edit, verify, and review — respecting architecture seams without overbuilding.

Delivery

Replan the next phase from shipped work

Use plan-next after a phase ships to propose Draft milestones for the next increment based on what actually shipped.

Governance

Pressure-test plans and terminology

Use $grill-with-docs when ambiguity, scope creep, or naming drift threatens trust in the docs — including architecture seams.

Alignment

Keep context synchronized

Use $context to keep docs, skills, agent rules, README, and user-journeys.html coherent after decisions settle.

The harness is strongest when it is used as a strict workflow spine — not as a replacement for the actual product codebase. Bootstrap determines the MVP boundary; architecture seams keep future work clean; plan-next reshapes the next increment from real results.

Key Artifacts

Startup contract

AGENTS.md defines non-negotiable operating rules, auto-triggers $context after settled decisions, and is where project-specific command gaps should be recorded.

Product backbone

docs/PRODUCT.md and docs/CONTEXT.md become product-specific during bootstrap and stay durable afterward.

Architecture seams

docs/ARCHITECTURE.md documents current structure, approved seams, and deferred architecture. Lightweight — updated by $context when seams open or close.

Execution evidence

.agent-trace/ holds local traces, while specs, milestones, and reviews preserve the durable summary.

Maintenance Note

Keep this file updated when the repo gains a new user-facing workflow, when a current journey changes its first step, or when the capability surface moves enough that the visible product story should change with it.
